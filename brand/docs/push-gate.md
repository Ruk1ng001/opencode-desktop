## 2. 推送决策

**结论：不推送。**

本轮（US-013 复检）逐条**实测**了各门槛，不引述历史结论：

- **G1 双包 typecheck**：从 `BASE_SHA` 干净基线重放全部补丁后真跑，desktop rc=0 / app rc=0。✅
- **G2 补丁 0 冲突重放**：磁盘 9 补丁（01-08 + 10）逐个 `git apply --check` + apply 全 `OK+APPLIED`，16 文件变更，无冲突。✅
- **G3 CI 4 平台矩阵**：`release.yml` `package` job 矩阵解析实测 4 项（macos-latest / macos-15-intel / windows-latest / ubuntu-latest）。✅
- **G4 敏感值不入库**：跟踪文件扫描无 `sk-` 明文，`brand.json` 唯一变动为 `defaultLocale:"zh"`（非密钥）。✅

未过的两项：

- **G6（UI 审查 US-012.5）— 硬阻塞**：PRD 中该 story 的 8 条 AC 仍全为 `[ ]`，progress.md 无任何 US-012.5
  记录，UI 审查这一环从未发生。它是 PRD 优先级 12、排在 US-013 之前的前置 story，未完成即门禁不通过。
- **G5（自动更新三平台实机）— 证据不足**：PRD 元数据已把 US-011 标 `done`，但其指定的实机结论表
  （`self-update-e2e.md` B.4）四行仍全「待验」、`TODO.md` 仍标「实机待跑」。无头 aarch64 服务器无 GUI /
  Electron 运行时 / 发布 token，物理上无法做真机装旧→更新→肉眼确认，实机证据未回填。仅凭 PRD 状态位
  单方认定「已过」会违反 US-011 确立的诚实原则（不伪造实机结论），故记为「证据不足」而非「已过」。

按 AC#5 / FR-17，任一门槛未过即不执行推送。技术门槛（G1/G2/G4）本轮已实测全绿、G3 配置层就绪；
G6 需先完成 US-012.5、G5 需回填三平台实机结论表，两者齐备后方满足全部推送条件。

## 3. 满足条件后的推送流程（G5/G6 通过后执行）

推送遵循分支安全（AC#3）与零人工冲突（AC#2 / FR-16）：

1. **不直接推 `main`**（除非明确授权）。先创建特性分支：
   ```sh
   git switch -c release/cx-<版本>   # 或 feat/us-013-push
   ```
2. **只暂存本轮真正变更的文件**，隔离会话前无关脏改动，绝不 `git add -A`。
   尤其注意父仓库常年存在的会话前脏文件（`brand/README.md` / `brand/brand.json` /
   `brand/patches.manifest` / `patch 02·04·10` 等半完成迁移态），提交前逐一核对归属。
3. 提交后推特性分支并设上游跟踪：
   ```sh
   git push -u origin release/cx-<版本>
   ```
4. 走现有 CI 自动合并发布流程（不手动 resolve 冲突）。**若跟随上游 / 发布过程出现人工
   冲突，视为设计缺陷（FR-15/FR-16）**：先把引发冲突的补丁重构为不冲突形态（迁独立文件 /
   改运行期兜底），再重推——不手动 resolve 后了事。
5. 发布产物齐全性以 `release-acceptance.md`（四平台矩阵）与 `self-update-e2e.md`
   （三平台自更新载体 + 元数据）为准；三平台自更新实机结论表四行全「通过」才把
   OLD/NEW 作为正式发布推送。
